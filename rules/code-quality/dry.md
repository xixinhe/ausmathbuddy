# DRY — Don't Repeat Yourself

每一份知识/逻辑在系统中应该只有一个唯一、明确的表达，避免重复。

## 规则

- 同样的逻辑出现两次以上，考虑提取成函数/模块；但不要为了"可能重复"而提前抽象（参考 [[kiss]]，三行相似代码不算重复）。
- 常量、配置项、文案只在一个地方定义，其他地方引用，不要复制粘贴。
- 修改 bug 或逻辑时，检查是否有其他地方存在同样的代码副本，一并确认是否需要同步修改。
- 重复的不只是代码，还包括重复的知识：比如同一个业务规则在前端和后端各写一遍、同一份数据结构在多处手动同步。

## 边界

- 表面相似但语义不同的代码不算重复，不要为了消除"看起来像"的重复而强行合并，导致耦合不相关的逻辑（过度抽象比重复更难维护）。
- 抽象前先确认重复会长期存在，而不是巧合性的相似。
- 与 [[kiss]] 冲突时，先看是否引入了不必要的间接层——简单的重复有时比复杂的抽象更易读。
