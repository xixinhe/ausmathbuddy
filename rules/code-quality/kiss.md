# KISS — Keep It Simple, Stupid

用最简单、直接的方式解决问题，拒绝不必要的复杂度。

## 规则

- 优先选择直白的实现，而不是"聪明"但难懂的写法。
- 不要为了应对假设中的未来需求而提前设计抽象层。
- 能用一个简单函数解决的问题，不要拆成多个类/模块。
- 减少条件分支和嵌套层级，逻辑尽量线性、易读。
- 命名清晰直接，代码本身就是文档，减少额外注释的依赖。
- 每次改动只做当前任务需要的事，不顺手重构无关代码。

## 反例

- 为只有一种实现的接口提前定义抽象基类/工厂模式。
- 用配置项、开关去覆盖根本不会发生的场景。
- 一个函数拆成五层调用，只是为了"看起来专业"。
