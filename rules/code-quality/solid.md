# SOLID 原则

面向对象/模块化设计的五个基本原则，用来保持代码可维护、可扩展。

## S — 单一职责原则（Single Responsibility Principle）

一个模块/类/函数只应该有一个改变的理由。

- 把"数据获取""业务逻辑""展示渲染"分开，不要揉在一起。
- 如果一个函数名字里出现"and"，通常说明它做了不止一件事。

## O — 开闭原则（Open/Closed Principle）

对扩展开放，对修改关闭。

- 新增功能时，优先通过新增代码（新函数/新分支/新模块）实现，而不是大改已经稳定、被多处依赖的旧逻辑。
- 但不要为此过度设计插件系统——先满足 KISS，再考虑扩展性。

## L — 里氏替换原则（Liskov Substitution Principle）

子类型必须能够替换其基类型，而不破坏程序正确性。

- 子类/实现不应该削弱父类/接口的承诺（比如抛出父类不会抛的异常、返回不兼容的类型）。

## I — 接口隔离原则（Interface Segregation Principle）

不要强迫调用者依赖它们用不到的接口。

- 拆分大而全的接口/参数对象，按实际使用场景提供小而专注的接口。

## D — 依赖倒置原则（Dependency Inversion Principle）

高层模块不应该依赖低层模块的具体实现，二者都应该依赖抽象（接口/约定）。

- 业务逻辑不要直接依赖具体的 DOM 操作、具体的存储实现，而是通过一层约定好的接口调用。
- 便于替换实现（比如把 localStorage 换成远程 API）而不改动业务代码。
