# 10-Point Readability Checklist

每次提交 PR / 每次 agent 输出代码后，逐条自查以下 10 点，不满足的要修正后再交付。

1. **命名清晰**：变量、函数、类名能准确表达意图，不需要看实现才懂。
2. **函数职责单一**：每个函数只做一件事，能用一句话说清楚它的作用（[[solid]] S）。
3. **无重复逻辑**：没有可以合并的重复代码/知识（[[dry]]）。
4. **无冗余复杂度**：没有为假设中的未来需求提前设计的抽象、开关、配置（[[kiss]]）。
5. **嵌套/分支合理**：条件分支和嵌套层级不过深，必要时提前 return 简化结构。
6. **注释只在必要处**：没有解释"做了什么"的废话注释，只在解释"为什么"这样做时才有注释。
7. **一致性**：命名风格、代码组织方式与项目现有代码保持一致，没有引入新的随意风格。
8. **无死代码**：没有遗留的调试代码、注释掉的旧代码、未使用的变量/导入。
9. **改动范围克制**：只改了任务需要的部分，没有夹带无关的重构（除非是 [[almo]] 允许的顺手小优化）。
10. **可独立理解**：不需要额外上下文（比如翻看之前的对话）就能看懂这段代码在做什么。

## 使用方式

- PR 提交前：过一遍这 10 点，逐条打勾。
- Agent 每次产出代码后：在给用户的总结前，先用这份清单自查一遍，发现问题直接修正，而不是等用户指出。
