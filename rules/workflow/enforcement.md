# 规则执行反馈机制

当 LLM（agent）的输出违反了 `rules/` 中的任一规则（[[kiss]]、[[solid]]、[[dry]]、[[almo]]、[[readability-checklist]]）时，按以下步骤处理，而不是简单指出错误就结束。

## 步骤

1. **定位违反的具体规则**：明确是哪一条、哪一句，而不是笼统地说"不够好"。
2. **把规则写得更具体**：如果违反的原因是规则本身太抽象、留有解释空间，回到对应的规则文件，补充更具体的描述或反例，让它下次不会被误解。
   - 例如 [[kiss]] 里"不要提前设计抽象层"太笼统，可以补充"具体到：只有一种实现时不要定义接口/抽象类"。
3. **必要时加一条 inline comment**：如果规则已经足够具体，但 LLM 在某一处具体代码上判断失误，就在那一行代码旁加一条简短的 inline 提示（说明违反了哪条规则、期望怎么改），而不是重写一大段说明。
4. **重试**：基于更新后的规则或 inline comment，让 LLM 重新生成/修改这部分输出。
5. **回验**：确认新输出确实解决了问题，且没有引入新的违反项（用 [[readability-checklist]] 过一遍）。

## 原则

- 优先修规则，而不是每次口头重复同样的纠正——规则修好了，以后同类问题不会再犯。
- inline comment 只用于一次性的、局部的判断失误，不用来代替规则本身。
- 不要因为一次违反就大幅重写规则文件，只补充刚好够用的具体化描述（呼应 [[kiss]]）。
