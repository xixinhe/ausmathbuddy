# Changelog 记录规则

每次完成一个功能，或者一份 PRD 的状态发生流转（[[adlc]]），都必须在项目根目录的 `CHANGELOG.md` 里补一条记录，不能只在对话里口头总结完就算完成。

## 触发时机

- 完成一个可交付的功能/模块（比如某个页面、某个逻辑跑通并验收通过）。
- PRD 文档状态变化（Draft → Review → Approved → Active → Deprecated，见 [[adlc]]）。
- 不需要为每一次小的代码修改都记录，只记"完成了什么"这个级别的节点，避免 CHANGELOG 被琐碎条目淹没。

## 记录格式

在 `CHANGELOG.md` 最上面追加一条（新记录在最上面，倒序时间线）：

```
## 2026-07-05
- [Feature] 完成 xxx 功能：一句话说明做了什么、影响范围
- [PRD] docs/prd.md: Draft → Review
```

- 日期用绝对日期（YYYY-MM-DD），不用"今天""昨天"。
- 类型前缀用 `[Feature]` / `[PRD]` / `[Fix]` 区分条目性质。
- 每条一句话，说清"做了什么"，不用展开实现细节（细节看 commit/代码本身）。

## 原则

- 记录动作和交付本身是同一次任务的一部分，不算完成就等于没做完（呼应 agent 完成任务前的自查清单）。
- 不为了记录而记录：口头讨论、还没验收通过的半成品不写入 CHANGELOG。
